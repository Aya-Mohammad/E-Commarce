# High-Performance E-Commerce Backend Engine 🚀
### محرك المعالجة عالي الأداء لمنظومة التجارة الإلكترونية

هذا المشروع هو عبارة عن محرك معالجة خلفي (Backend Engine) مبني باستخدام إطار العمل **Laravel (PHP)**، ومصمم خصيصاً للتعامل مع الأحمال المتزامنة العالية (High Concurrency) وحل المشاكل المعمارية الناتجة عن تدفق آلاف الطلبات في أجزاء من الثانية. تم تطوير المشروع كجزء من المتطلبات العملية لمادة **البرمجة المتوازية (Parallel Programming)** في جامعة دمشق - كلية الهندسة المعلوماتية لعام 2026.

---

## 🛠️ الزمام الهندسية والتقنيات المستخدمة (Tech Stack)
* **Framework:** Laravel 11 (PHP 8.2+)
* **Database:** MySQL (مع تفعيل ACID Transactions و Pessimistic Locking)
* **Caching & Distributed Locks:** Redis
* **Asynchronous Processing:** Laravel Queues & Jobs (Database/Redis driver)
* **Benchmarking & Stress Testing:** Apache JMeter

## 🏗️ البنية المعمارية ومنهجية الكود (Architecture & Code Structure)

تم عزل وتغليف المنطق البرمجي الخاص بمعالجة الطلبات بالكامل داخل طبقة الخدمات، وتحديداً ضمن الصنف `App\Services\OrderService`. 

لتسهيل عملية التقييم وإثبات الفروقات الهندسية للعمليات المتزامنة، تم هيكلة الكود المصدري داخل الدالة الحرجة `placeOrder()` بأسلوب اختبار المقارنة (**A/B Testing Structure**)، حيث تم تقسيم الكود إلى قسمين رئيسيين:

### 1. الكتل الاختبارية المعلقة (`Before Handling - Commented Blocks`)
تم الاحتفاظ بالنسخ البرمجية السابقة (التي تمثل النظام قبل التحسين) على شكل كتل برمجية معلقة. كل كتلة مخصصة لمحاكاة مشكلة هندسية محددة:
* **Test NFR 1 (Locking):** يحاكي غياب الأقفال وحدوث الـ Race Condition والـ Overselling.
* **Test NFR 2 (Capacity Control):** يحاكي غياب التحكم بالسعة وانعدام إدارة الموارد.
* **Test NFR 3 (Queue):** يحاكي المعالجة المتزامنة للمهام الثقيلة بدون طوابير (توليد الفواتير وإرسال الإيميلات الفوري الذي يسبب بطء الاستجابة).
* **Test NFR 4 (Batch):** يحاكي الإدراج الفردي المجهد لقاعدة البيانات (`Row-by-Row insertion`).

> 💡 **ملاحظة للمُقيّم:** يمكنك تفعيل أي من هذه الكتل البرمجية عبر إلغاء التعليق (`Uncomment`) لإعادة إنتاج المشاكل واختبار الاختناقات (`Bottlenecks`) وقياس الأداء بنفسك عبر JMeter.

### 2. النسخة النهائية المُحسنة الفعالة (`Improved Code`)
وهي الدالة النشطة حالياً في النظام، وتعمل بالتسلسل الهندسي التتابعي التالي لحل جميع المتطلبات غير الوظيفية (NFRs):
1. **حماية الخادم (Global Rate Limiting):** عبر `RateLimiter::attempt` كخط دفاع أول ضد طفرات الطلبات الهائلة لمنع انهيار الخادم.
2. **التحكم بالسعة (Capacity Control):** التحقق من حجم السلة عبر الذاكرة المؤقتة (`Cache::remember`) قبل فتح معاملات مجهدة مع قاعدة البيانات.
3. **منع التكرار (Distributed Redis Lock):** حظر تقديم الطلب المزدوج لنفس المستخدم في نفس اللحظة.
4. **المعاملات الآمنة ومنع القفل الميت (ACID & Deadlock Prevention):** استخدام الترتيب الرياضي `orderBy('id')` متبوعاً بقفل الحجز التشاؤمي `lockForUpdate()` لضمان معالجة دقيقة للمخزون ومنع الـ Deadlock التنافسي.
5. **إدراج الدفعات (Batch Insertion):** تقليل رحلات الاتصال (`Round-Trips`) بقاعدة البيانات لادراج عناصر الطلب دفعة واحدة.
6. **إبطال الذاكرة المؤقتة (Cache Invalidation):** تصفير بيانات السلة والكاش القديم فوراً لضمان اتساق البيانات المطلقة.
7. **المعالجة غير المتزامنة (Async Queues):** ترحيل المهام الخلفية الثقيلة (`SendOrderConfirmationJob` و `GenerateInvoiceJob`) للعمل في الخلفية لتحقيق هبوط حاد في زمن الاستجابة للمستخدم.
